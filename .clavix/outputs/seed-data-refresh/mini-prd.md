# Requirements: Seed Data Refresh Script

*Generated from conversation on 2026-03-17*

## Objective
Create a script that automates refreshing the three Cincinnati-sourced seed data files in `src/imageset_generator/data/`, intended as a release management step to ensure packaged seed data stays current.

## Core Requirements

### Must Have (High Priority)
- [HIGH] Script at `scripts/refresh-seed-data.sh` that automates the full refresh workflow
- [HIGH] Refreshes exactly three files: `ocp-versions.json`, `ocp-channels.json`, `channel-releases.json`
- [HIGH] Leverages the app's existing refresh API endpoints (no direct Cincinnati calls)
- [HIGH] Flow: start app locally -> hit refresh endpoints -> copy runtime cache to `src/imageset_generator/data/` -> stop app
- [HIGH] Validation: verify refreshed data is not a regression (e.g., new data has at least as many versions/channels as the current seed data)
- [HIGH] amd64 architecture only

### Should Have (Medium Priority)
- [MEDIUM] Clear error messages when validation fails, showing what regressed (e.g., "versions dropped from 23 to 1")
- [MEDIUM] Non-zero exit code on validation failure so CI/release scripts can gate on it

### Could Have (Low Priority / Inferred)
- [LOW] Summary output showing what changed (new versions added, channels updated)

## Technical Constraints
- **Runtime:** Bash script using `curl` to hit the app's REST endpoints
- **App startup:** Must start the Flask app, wait for readiness, then proceed
- **Python env:** Must use `.venv/bin/python` or `uv` (never system python, per project convention)
- **File paths:** Runtime cache at `data/` (project root), seed data at `src/imageset_generator/data/`
- **Architecture:** amd64 only — no arch-scoped filenames needed

## Architecture & Design
- **Pattern:** Thin shell script orchestrating the existing app
- **Key Decision:** Reuse existing refresh endpoints rather than calling Cincinnati directly — avoids duplicating discovery logic

## User Context
- **Target Users:** Project maintainers performing a release
- **Primary Use Case:** Before cutting a release, run this script to ensure seed data ships with current OCP version/channel information
- **User Flow:** `scripts/refresh-seed-data.sh` -> review changes -> commit updated seed files

## Edge Cases & Considerations
- App may fail to start (missing deps, port conflict) — script should detect and exit cleanly
- Cincinnati may be unreachable — refresh endpoints will return cached/stale data; validation should catch if data didn't actually update
- Partial refresh failure (e.g., versions refresh succeeds but channels fails) — script should report which files failed
