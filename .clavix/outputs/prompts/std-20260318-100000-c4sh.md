# Improved Prompt

Objective: Make the runtime JSON cache resilient to transient API failures so stale
or partial data never permanently shadows the packaged seed data.

Context:
- The app has a two-tier data strategy: runtime cache files in `data/` take priority
  over packaged seed data in `src/imageset_generator/data/`.
- `get_data_read_path()` in constants.py returns the runtime file if it exists,
  regardless of its age or completeness.
- Cache files are written by `/api/*/refresh` endpoints which call the Cincinnati API
  and `opm render`/`skopeo inspect`.
- If an API call partially succeeds (e.g., only 4.16 is reachable), the refresh
  endpoint writes a cache file with incomplete data. This file then permanently
  shadows the packaged seed data that has 23 versions.
- There is no file-level TTL, staleness check, or completeness validation on
  cached JSON files.

Requirements:
1. Validate cache freshness: add a file-level TTL (e.g., configurable max age) to
   `get_data_read_path()`. If a runtime cache file is older than the TTL, fall through
   to packaged seed data.
2. Validate cache completeness: before accepting a runtime cache file, verify it
   meets a minimum plausibility threshold (e.g., ocp-versions must have more than
   N releases). If it fails, fall through to seed data and log a warning.
3. Atomic cache writes: write to a temp file first, then rename, so a crash
   mid-write never leaves a truncated JSON file.
4. Preserve the existing in-memory TTL cache in discovery.py (1-hour session TTL) —
   these changes target the persistent file cache only.

Files to modify:
- src/imageset_generator/constants.py — get_data_read_path(), add TTL + validation
- src/imageset_generator/app.py — refresh endpoints, atomic writes
- tests/ — unit tests for cache staleness, completeness, and atomic write behavior

Technical constraints:
- Python, Flask backend
- No new dependencies unless justified
- Cache files are JSON with `timestamp`, `source`, and payload fields

Success criteria:
- A stale or partial cache file does NOT prevent the app from showing seed data
- A fresh, complete cache file still takes priority over seed data
- Refresh endpoints write atomically (no partial files on disk)
- All new logic has unit test coverage
